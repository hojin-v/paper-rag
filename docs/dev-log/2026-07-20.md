## 사고 기록: 로컬 dev DB 카탈로그 손상 및 복구

검증 과정에서 `docker compose up -d postgres`를 실행해 이 프로젝트의 실제 운영 DB
(`paper-rag-postgres`, 포트 5433 — docker-compose가 아니라 별도로 기동되어 있던 컨테이너)와
**동일한 이름의 Docker 볼륨(`paper-rag_pgdata`)을 공유하는 두 번째 Postgres 컨테이너**를 실수로
만들었다. 이 컨테이너를 정리(`docker compose rm`)하는 과정에서 실제 컨테이너가 크래시했고, 재시작 후
반복적인 WAL 플러시 실패(`xlog flush request ... is not satisfied`)와 `review_documents` 관련
인덱스가 부모 테이블 없이 카탈로그에만 남는(`DROP INDEX`도 실패, OID 파일 자체가 사라짐) 실제
카탈로그 손상이 발생했다.

- **확인된 안전 범위**: `papers`/`paragraphs`/`keywords` 등 기존 9개 테이블은 `VACUUM ANALYZE`로
  전부 정상 확인(손상은 이번 세션에서 새로 만든 `review_documents` 객체에 한정).
- **복구**: 손상되지 않은 전체 테이블을 `pg_dump --exclude-table=review_documents`로 백업 →
  컨테이너·볼륨 삭제 후 동일 설정으로 재생성 → 백업 복원 → 마이그레이션 0004 재적용 →
  backfill 재실행. 최종적으로 papers=5/paragraphs=70/keywords=17 등 사고 이전과 동일한 개수로 복원,
  `review_documents` 27건 정상 backfill, FK 무결성 검증(고아 참조 0건) 완료.
- **교훈**: 로컬 환경에 프로젝트 DB가 docker-compose 밖에서(수동 `docker run`) 떠 있을 수 있다 —
  `docker compose up`으로 서비스를 개별 기동하기 전에 `docker ps`로 동일 이름의 컨테이너가 이미
  떠 있는지, 같은 볼륨을 쓰는지 반드시 먼저 확인할 것.

## 2차 세션 — PyMuPDF(AGPL-3.0) 제거

`docs/reports/assessments/2026-07-20-oss-deployment-risk-model-tradeoff.md`가 "높음" 위험·최우선
권고로 지적한 PyMuPDF를 permissive 라이선스 라이브러리로 전면 교체했다(13개 파일: 운영 코드 6개,
테스트 7개).

- **운영 코드**: `paddle_backend.py::_render_pdf_pages`(핵심 OCR 경로)·`review/service.py`의
  `_render_pages`/`_write_ocr_crops`·`triage.py::classify_pdf`·`readiness.py`의 임포트 프로브를
  `pypdfium2`(Apache-2.0/BSD-3-Clause, Google PDFium 바인딩)로 교체. `collect/smoke.py`의 페이지
  복사는 `pypdf`(BSD-3-Clause)로, 진단 전용 `simple_backend.py`(`SimplePyMuPDFBackend` →
  `SimpleTextLayerBackend`로 리네임)는 `pdfplumber`(MIT)로 교체.
- **좌표계 호환성**: pdfplumber의 `top`/`bottom`은 이미 좌상단 원점(y 아래로 증가) 관례를 따라
  PyMuPDF·기존 `LayoutBlock.bbox` 규약과 변환 없이 호환된다. pypdfium2는 렌더링(`scale` = dpi/72,
  PyMuPDF의 `Matrix(scale,scale)`과 동일 의미)만 쓰고 bbox 추출에는 쓰지 않아 좌표계 이슈가 없다.
  단, pypdfium2는 PyMuPDF의 `clip=Rect` 임의 영역 렌더링을 지원하지 않아, OCR crop
  export(`_write_ocr_crops`)는 페이지 전체를 페이지당 1회만 렌더링해 캐시하고 PIL로 crop하는
  방식으로 재구현.
- **테스트 픽스처**: `tests/pdf_fixtures.py`(신규)에 reportlab(BSD) 기반 `PdfBuilder`를 만들어
  `pymupdf.open() → new_page() → insert_text()` 패턴을 좌상단 원점 좌표 그대로 대체(내부에서
  `canvas_y = height - y`로 변환). `tests/conftest.py`가 `tests/` 자체를 `sys.path`에 추가하도록
  갱신해 `tests/integration/`의 테스트에서도 `pdf_fixtures`를 바로 임포트 가능하게 함.
- **검증**: `pymupdf` 패키지를 venv에서 완전히 제거한 뒤 `pytest -q` 168 passed / 2 skipped로
  재확인(잔존 의존 없음 증명). 실제 수집된 22페이지 논문 PDF·200DPI 기준 렌더링 실측:
  pypdfium2 0.069s/page vs 기존 PyMuPDF 0.050s/page(약 20ms/page 느려짐, OCR 추론 시간 대비
  무시 가능), 페이지 PNG 파일 크기는 약 47% 증가(압축 특성 차이로 추정, 추가 최적화 안 함).
- 문서 갱신: `DESIGN.md`·`docs/guide/01-environment.md`·`docs/guide/04-ingest-pipeline.md`·
  `docs/guide/10-production-readiness.md`·발표자 노트·위험평가 문서·코드베이스 읽기 가이드의
  PyMuPDF 언급을 전부 교체/해결 표시로 갱신.

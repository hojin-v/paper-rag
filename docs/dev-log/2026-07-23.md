## CI/CD 실전 검증 중 발견한 문제 3건

첫 실전 push에서 GitHub Actions(`ci.yml`/`deploy.yml`)를 실제로 동작시켜보며, 로컬
테스트로는 드러나지 않는 문제 3건이 연달아 나왔다("테스트는 통과하지만 실제로
돌려보면 다른 문제"의 또 다른 사례).

1. **CI 의존성 설치 실패**: `requirements-lock.txt`가 고정한 `torch==2.8.0+cpu`는
   기본 PyPI가 아니라 `download.pytorch.org/whl/cpu` 전용 인덱스에만 있는 빌드다.
   Dockerfile은 이미 이 인덱스를 알고 있었지만 `ci.yml`은 몰랐다. → 같은 방식으로
   먼저 설치하도록 수정.
2. **배포 스크립트가 worker/ui를 재기동에서 누락**: `docker-compose.yml`의
   `worker`/`ui` 서비스에 `profiles`가 붙어 있어, 서비스명을 명시하지 않은
   `docker compose up -d`/`build`는 이 둘을 조용히 건너뛴다. `stop`은 서비스명을
   명시해 정상 작동했지만 재기동만 빠뜨려, 첫 실배포에서 `worker`가 새 이미지로
   올라오지 못하고 멈춰 있었다(`ui`는 애초에 생성된 적도 없음). → build/up/롤백
   세 곳 모두 서비스명을 명시하도록 수정.
3. **배포 스크립트가 개발 디렉터리를 조용히 훼손**: `PAPERRAG_DEPLOY_DIR`가 곧
   개발자의 작업 디렉터리이기도 한데, 배포 스크립트의 `git checkout --force`가
   그 시점의 미커밋 변경을 경고 없이 버리고(실제로 스크립트 자신을 고치던 중
   겪음 — 커밋 전 수정분이 사라짐) HEAD를 분리(detached) 상태로 남겼다. →
   dirty 상태면 stash로 보존하고, 스크립트 종료 시 원래 브랜치로 복귀하도록 수정.

세 건 모두 실배포로 재검증했고, 이제 `main` push → CI(클라우드) → CD(셀프호스트
러너, 4개 앱 서비스 전부 재기동) → 헬스체크까지 전체 파이프라인이 실제로
정상 동작함을 확인했다.

## 버그: OCR 실행 실패 2건 (실 검수 작업 중 발견)

1. **"daemonic processes are not allowed to have children"**: `_run_isolated_paddle`가
   Paddle 연산을 격리하려고 항상 별도 프로세스를 spawn했는데, 비동기 OCR 경로가
   이번에 처음 실제 Celery 워커에서 실행되며 — Celery 워커 프로세스 자체가 데몬
   프로세스라 그 안에서 또 자식 프로세스를 만들 수 없어(Python 자체 제약)
   즉시 실패했다. → 이미 분리된 워커 프로세스 안에서 호출된 경우 spawn 없이
   같은 프로세스에서 직접 실행하도록 분기 추가. 실제 운영 문서(237블록, 표
   다수)로 재검증 완료.
2. **"Timeout reading from socket"**: 위 OCR 작업이 무거운 작업 슬롯(동시 1개
   제한)을 20분 넘게 붙잡고 있는 동안, 재시도한 다른 요청이 슬롯을 최대 60초
   (`heavy_task_semaphore_wait_seconds`)까지 기다려야 하는데 Redis 클라이언트
   소켓 타임아웃은 5초로 고정돼 있어, 정상적인 "슬롯이 꽉 찼다" 메시지 대신
   날것의 소켓 타임아웃 에러가 노출됐다. → 소켓 타임아웃을 대기 시간보다 항상
   크게 설정. 실제 Redis로 슬롯 경합을 재현해 정상적으로 `HeavyTaskBusyError`가
   뜨는 것과 토큰 유실이 없는 것을 확인.

## 신규 기능: 로깅 · LLM 호출 관찰

위 두 버그의 원인 파악이 Celery 기본 트레이스백에만 의존했던 것을 계기로, 애플리케이션

Postgres+ClickHouse+Redis+MinIO 필요), 기존 Postgres에 기록 + 자체 뷰어 방식을
선택했다(`docs/adr/0003-observability-approach.md`).

  `prompt_eval_count`/`eval_count`를 그대로 옮겨 별도 계측이 필요 없었다.
- `GET /observability/llm-calls` 조회 뷰어(review 라우터에 편입, 기존 인증 재사용).

# 발표자 설명 및 평가자 예상 Q&A

이 문서는 발표 화면에 넣지 않는 설명이다. `PRESENTATION.md`의 슬라이드 번호와 동일한 순서로 구성했다.
각 항목은 **발표자 설명**과 **평가자 질문·답변**을 분리했다.

## 1. 온프레미스 논문 분석 RAG

### 발표자 설명

이 시스템은 논문 원문을 외부 서비스로 보내지 않고 내부망에서 분석한다. 비정형 PDF를 본문 단락,
표, 메타데이터, 키워드와 벡터로 구조화한 뒤 자연어 검색과 엑셀 결과물로 제공한다.

### 평가자 질문·답변

**Q. 일반적인 PDF 검색과 무엇이 다른가?**
A. 파일명이나 전체 텍스트만 검색하지 않는다. 논문의 구조를 복원하고 본문 단락·키워드·표를 각각
저장하며, 사용자가 추출 영역과 OCR 원문을 직접 검수할 수 있다.

**Q. 왜 온프레미스가 필요한가?**
A. 사내 논문, 유료 논문, 미공개 연구자료가 외부 API로 전송되는 것을 방지하고 모델·DB·로그를 내부에서
통제하기 위해서다.

## 2. 문제

### 발표자 설명

PDF는 화면상 문서이지 검색용 데이터가 아니다. 특히 2단 조판, 스캔, 표, 머리말과 참고문헌 때문에
단순 텍스트 추출만으로는 본문 순서와 의미가 깨진다. 또한 자동 추출 결과를 확인할 방법이 없으면 DB에
오류가 누적된다.

### 평가자 질문·답변

**Q. PDF 전체를 그대로 임베딩하면 안 되는가?**
A. 문서 전체 벡터는 세부 근거를 찾기 어렵고 참고문헌·캡션·머리말이 검색을 오염시킨다. 본문 단락으로
분리해야 검색 결과를 원문 근거와 함께 제시할 수 있다.

## 3. 두 사용자 경험

### 발표자 설명

등록자는 분석 품질을 책임지고, 검색자는 결과 활용에 집중한다. 등록 화면은 과정의 투명성이 중요하고,
검색 화면은 핵심 안내와 다운로드가 중요하다.

### 평가자 질문·답변

**Q. 모든 영역을 사람이 검수해야 하는가?**
A. 아니다. 전체 승인 기능을 제공하고, 실제 운영에서는 confidence가 낮은 영역을 우선
검수한다. 검수 데이터는 이후 레이아웃/OCR 학습데이터로 재사용한다.

**Q. 교정 전 데이터가 검색 DB에 들어갈 수 있는가?**
A. 검수 문서와 검색 DB 적재를 분리했다. 사용자가 적재를 실행할 때 승인·교정된 결과가 파이프라인에
전달되고 제외 영역은 저장하지 않는다.

## 4. 전체 아키텍처

### 발표자 설명

등록과 검색은 PostgreSQL+pgvector를 중심으로 만난다. 별도 Vector DB를 추가하지 않아 메타데이터 조인,
벡터 검색, 삭제와 백업을 한 저장소에서 관리한다. 모델은 모두 함수 내부 지연 로딩 또는 별도 서버로
분리돼 있다.

### 평가자 질문·답변

**Q. 왜 전용 Vector DB가 아닌 pgvector인가?**
A. 핵심 검색이 정확 키워드 SQL과 벡터 유사 검색의 결합이고, 결과 엑셀이 논문·단락·표를 다중 조인한다.
현재 예상 규모에서는 단일 트랜잭션과 백업이 운영상 더 유리하다.

**Q. 데이터가 커지면 확장 가능한가?**
A. 수십만~백만 단락은 HNSW와 배치 적재로 처리하고, 실제 검색 지연과 인덱스 크기가 기준을 넘을 때
Qdrant나 Milvus 분리를 검토한다. 저장소 분리는 측정 후 결정한다.

**Q. 현재 업로드가 production 수준의 비동기 처리인가?**
A. 아니다. OCR과 적재는 현재 HTTP 요청 안에서 실행된다. Celery worker와 공유 볼륨은 구성했지만 UI 적재
버튼과 작업 상태 조회가 연결되지 않았다. 내부 단일 사용자 MVP 이후 작업 제출·polling API가 필요하다.

## 5. 저장하는 정보

### 발표자 설명

검색 가치가 있는 본문과 표는 유지하고 그림, 독립 수식, 참고문헌과 부록은 제외한다. 단락마다 원문을
보존하기 때문에 LLM 정제·요약 결과가 잘못돼도 원문과 비교할 수 있다.

### 평가자 질문·답변

**Q. 그림과 수식을 모두 버려도 되는가?**
A. 현재 범위는 텍스트 중심 논문 검색이다. 원본 PDF와 레이아웃 좌표는 보존하므로 향후 멀티모달 요구가
생기면 재분석할 수 있다. 지금은 불필요한 모델과 저장 비용을 줄이는 선택이다.

**Q. 용어설명은 어떻게 제외하는가?**
A. 현재는 독립된 용어설명 클래스를 두지 않았다. 실제 논문 평가셋에서 반복적으로 검색을 오염시키면
섹션명·블록 유형·LLM 관련도 판정을 조합한 명시적 필터를 추가해야 한다.

## 6. 검색 원칙

### 발표자 설명

정확한 키워드가 있으면 벡터 검색보다 우선한다. 검색어가 DB 키워드와 다를 때만 의미적으로 가까운
후보를 제시하고 사용자가 선택한다. 이 방식은 임의의 유사 결과가 바로 확정되는 것을 막는다.

### 평가자 질문·답변

**Q. 왜 처음부터 의미 검색만 사용하지 않는가?**
A. 논문 키워드는 전문용어의 정확성이 중요하다. 동일한 용어가 존재할 때는 SQL 정확 매칭이 빠르고
설명 가능하며, 미매칭 상황에서만 벡터 검색을 사용해야 오탐을 줄일 수 있다.

**Q. 회의록의 관련 5개는 관련 논문 5편을 의미하는가?**
A. 아니다. 논문에서 추출하는 대표 키워드 3~5개 요구가 중복 기재된 것이다. 검색 결과 형식은
대표 논문 1편과 연관 논문 1편이며, 엑셀에서도 이 두 논문의 메타데이터와 단락을 제공한다.

## 7. 결과물: 6시트 엑셀

### 발표자 설명

화면에는 결과를 빠르게 판단할 정보만 보여주고, 검토와 보고에 필요한 원문·요약·표는 엑셀로 제공한다.
대표 논문과 연관 논문을 동일한 구조로 비교할 수 있다.

### 평가자 질문·답변

**Q. 왜 웹 화면에서 모든 내용을 보여주지 않는가?**
A. 긴 원문과 표는 화면에서 비교하기 어렵고 사용자는 사내 검토·보고 자료로 재사용하려 한다. 엑셀은
필터, 복사, 주석과 오프라인 전달이 쉬워 현재 사용자 시나리오에 적합하다.

**Q. 엑셀 파일의 추적성은 어떻게 확보하는가?**
A. 검색 질의, 매칭 키워드, 매칭 방식, 점수와 선정 이유를 첫 시트에 기록한다. 운영 단계에서는 모델,
프롬프트와 데이터 버전도 함께 기록할 예정이다.

**Q. 모델·프롬프트 버전이 지금 DB에 기록되는가?**
A. 아직 아니다. 현재 재현성 공백으로 명시했으며 외부 논문 수집 전에 출처·라이선스·checksum과 함께
processing run manifest를 추가해야 한다.

## 8. CPU 중심 모델 라우팅

### 발표자 설명

입력 PDF의 텍스트 레이어 품질과 무관하게 모든 페이지를 이미지로 변환한다. 동일한 PP-StructureV3
레이아웃·OCR 경로를 적용한다. 표는 현재 좌표 영역의 OCR 토큰을 행·열 순서로 정렬해 보존하고, 정밀 셀
모델은 실제 표 TEDS가 기준에 못 미칠 때만 추가한다.

### 평가자 질문·답변

**Q. CPU만으로 현실적인가?**
A. 실행 자체는 확인했다. 200 DPI 합성 1페이지는 약 60~70초였지만 실제 2단 LayoutLMv2 첫 페이지는
약 5분 30초가 걸렸다. 긴 논문을 동기 응답으로 처리하는 것은 현실적이지 않으므로 작업 큐 기반 배치로
전환해야 하며, 페이지당 p50/p95는 실제 논문 10편에서 다시 측정한다.

**Q. 디지털 PDF의 텍스트 레이어를 사용하지 않는 이유는?**
A. 입력마다 텍스트 순서와 인코딩 품질이 달라 서로 다른 추출 결과가 만들어지는 것을 피하기 위해서다.
모든 PDF에 동일한 이미지·좌표·OCR 기준을 적용하면 검수 화면과 학습데이터 형식이 일관된다. 대신 처리
시간과 OCR 오탈자가 증가하므로 CER와 처리 시간을 반드시 측정한다.

**Q. 최소 서버 사양은?**
A. 순차 처리 기준 현대적 8코어/16스레드, RAM 32GB, NVMe 250GB가 최소다. PostgreSQL과 모델을 함께
운영하고 대량 적재하려면 16코어/32스레드, RAM 64GB, NVMe 1TB를 권장한다.

## 9. 모델 선택

### 발표자 설명

모든 PDF는 `PP-DocLayout-M`, `PP-OCRv5_mobile_det`, `korean_PP-OCRv5_mobile_rec` 조합을 사용한다.
표 구조 모델 파일도 조사·반입했지만 PP-StructureV3 정밀 셀 경로는 추가 대형 검출 모델을 요구하므로
기본 서비스에는 연결하지 않았다. Docling과 PyMuPDF 텍스트 파서는 비교·장애 진단용으로만 남긴다.

### 평가자 질문·답변

**Q. 하나의 거대 비전 모델로 처리하지 않는 이유는?**
A. CPU 환경과 폐쇄망 운영에서는 지연, 메모리와 모델 반입 부담이 크다. 레이아웃·OCR을 경량 조합으로
실행하고, 정밀 표 셀 모델은 실제로 필요한 영역에만 적용하는 편이 실패 원인을 추적하기 쉽다.

**Q. LLM 파인튜닝은 왜 지금 하지 않는가?**
A. 프롬프트 기준선과 골드 평가셋 없이 학습하면 개선 여부를 판단할 수 없다. JSON 유효율, 키워드 F1,
요약 환각률과 검색 recall을 먼저 측정하고 미달 태스크만 QLoRA 대상으로 삼는다.

## 10. 검증된 범위

### 발표자 설명

자동 테스트 78건과 별도로 PostgreSQL+pgvector, BGE-M3 CPU, Qwen2.5-7B Q4, Paddle CPU를 실제 기동했다.
CC BY LayoutLMv2 첫 페이지도 업로드·클릭 검수 API까지 통과했지만 약 5분 30초가 걸렸고 제목·Abstract와
본문 일부가 누락됐다. CC BY/CC BY-SA 논문은 66편·1,175페이지를 checksum 검증해 확보했다. 따라서
실행 가능성과 평가 입력 확보는 검증됐고 실제 논문 품질 합격은 아니라고 구분한다.

### 평가자 질문·답변

**Q. 테스트 78건이면 품질이 충분한가?**
A. 기능 회귀에는 의미가 있지만 논문 추출 품질을 보장하지는 않는다. 실제 디지털·2단 조판·스캔 논문 평가셋으로
mAP, CER, TEDS, 단락 F1과 검색 recall을 별도로 측정해야 한다.

**Q. 현재 가장 큰 검증 공백은?**
A. 실제 논문 PDF 평가셋과 운영급 CPU에서의 경량 LLM 지연·품질이다. 합성 PDF 성공을 운영 품질로
해석하지 않는다.

**Q. hash 임베딩으로 검색 데모를 성공시킨 것 아닌가?**
A. hash는 차원과 HTTP 계약을 시험하는 결정적 가짜 벡터일 뿐 검색 성능 근거가 아니다. 운영 compose는
BGE-M3 모드만 사용하고 `/ready`는 hash를 오류로 판정한다.

**Q. 지금 production 완료라고 말할 수 있는가?**
A. 아니다. 현재 머신의 런타임·모델과 `/ready`는 준비됐지만 실제 논문 품질, 비동기 처리, 인증·격리와
처리량 기준은 미완료다. 내부 단일 사용자 MVP를 실사용 평가할 수 있는 상태이지 외부 공개 production
완료 상태는 아니다.

## 11. 현재와 목표

### 발표자 설명

기능형 MVP와 실모델 실행 경로는 준비됐지만 실제 논문 수용 테스트가 남았다. 대표 1편과 연관
1편의 근거 단락을 이용한 업종 관련 생성 답변은 다음 범위다. 먼저 실제 논문 품질을 측정하고,
검색 평가셋을 만든 뒤 답변 생성과 인용 검증을 추가한다.

### 평가자 질문·답변

**Q. 업종 답변이 환각하지 않는다는 보장이 있는가?**
A. 답변 입력을 대표·연관 논문의 검색 근거 단락으로 제한하고 각 문장에 논문·단락 ID를 연결해야 한다.
근거에 없는 내용은 답하지 않도록 프롬프트와 후검증을 적용하며, 업종 분류표가 있다면 DB의 통제
어휘로 관리한다.

**Q. 다음 우선순위는 무엇인가?**
A. ① 실제 논문 10편 수용 테스트 ② 대표 키워드 3~5개와 검색 품질 측정 ③ 비동기 OCR 작업 연결
④ 대표·연관 논문 근거 기반 업종 답변 ⑤ 기준 미달 영역만 모델 확대·파인튜닝 순서다.

## 12. 기대 효과

### 발표자 설명

핵심은 모델 자체가 아니라 신뢰 가능한 논문 데이터 흐름이다. 추출 과정을 사람이 확인할 수 있고,
검색 결과가 원문 단락과 연결되며, 엑셀로 바로 활용되는 폐쇄망 연구 지원 기반을 제공한다.

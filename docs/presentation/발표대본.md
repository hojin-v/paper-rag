# 최종 발표 대본 — 발표자료.pptx (20장)

`발표자료.pptx`(20장) 전용 발표자 스크립트와 예상 Q&A. 슬라이드 순서와 1:1 대응한다.
**약 9~10분** 기준(슬라이드당 25~35초, 다이어그램·표는 조금 더, 소회는 가변).
성능 평가 두 장(14·15)의 상세 대본·Q&A는 `EVALUATION_NOTES_AND_QA.md`를 함께 본다.

---

### 1. 커버
**말할 것**: "온프레미스 논문 분석 RAG를 소개합니다. 비정형 PDF를 검수 가능한 지식과 엑셀로 바꾸는,
폐쇄망에서 동작하는 시스템입니다."

### 2. 문제 (기획 개요)
**말할 것**: "논문 PDF는 구조·품질이 제각각이고 본문·표·참고문헌이 뒤섞여 있어, 사람이 일일이 확인하고
정리해야만 검색에 쓸 수 있습니다. 저희 기획 의도는 이 정리를 자동화하고, 검색 결과엔 근거까지 붙여 바로
활용하게 하는 겁니다."
- **Q. 일반 PDF 검색과 뭐가 다른가?** A. 파일명·전문 검색이 아니라 문서 구조를 복원해 단락·키워드·표를 각각
  저장하고, 결과에 원문 근거와 생성 설명을 붙입니다.

### 3. 서비스 소개 — 두 사용자 경험
**말할 것**: "경험은 둘입니다. 등록은 논문을 올리면 자동 구조화되고 필요할 때만 사람이 보정해 적재하는
반자동 흐름이고, 검색은 자연어 질의로 대표·연관 논문과 근거를 받습니다. 각 단계가 실제 API로 연결됩니다."
- **Q. 모든 걸 사람이 검수하나?** A. 아니요. 신뢰도 낮은 부분만 우선 보정하고 전체 승인도 가능한 반자동입니다.

### 4. 대표 기능
**말할 것**: "핵심은 자동화입니다. 전 페이지 OCR·레이아웃 구조화, 단락·표·키워드 자동 추출과 요약, 정확
키워드 검색과 유사 제안, 근거 기반 관련도 설명, 엑셀 자동 생성 — 사람이 일일이 정리하던 일을 대신합니다."

### 5. 핵심 모델 — 검색(임베딩)과 생성(LLM)
**말할 것**: "두 모델이 RAG의 두 축입니다. 임베딩(BGE-M3)이 근거 단락을 찾고, LLM(Qwen2.5)이 그 근거로
답을 만듭니다. 모든 LLM 호출은 JSON 스키마를 강제하고 temperature 0으로 결정적 출력을 냅니다."
- **Q. 왜 하나의 큰 모델이 아니라 조합인가?** A. CPU·폐쇄망 환경에선 태스크별 경량 모델 조합이 실패 원인
  추적과 자원 관리에 유리합니다. 7B급도 Q4 양자화로 CPU 구동을 확인했습니다.

### 6. 프롬프트 설계 — 페르소나 + JSON 강제
**말할 것**: "태스크마다 좁은 역할을 명시하고 JSON 스키마를 강제합니다. '입력에 없는 내용 생성 금지'로
환각을 억제하고, 언어 오염이 감지되면 1회 재시도, 그래도 실패하면 폴백 또는 명시적 실패로 처리합니다."

### 7. 생성 단계 — RAG
**말할 것**: "대표·연관 논문이 정해지면 질의와 가장 가까운 근거 단락 하나를 골라, 그 단락만 근거로 '왜 이
논문이 관련 있는지'를 생성합니다. 단순 검색이 아니라 retrieval에 근거 제한 generation이 결합된 구조입니다."
- **Q. 결국 키워드 검색 아닌가?** A. retrieve한 근거 단락으로 generate하는 단계가 있습니다. 다만 여러 문서를
  종합하는 대화형은 아직 아니고 논문별 단답형 관련도 설명이라는 점은 과장 없이 밝힙니다.

### 8. 데이터 파이프라인 흐름
**말할 것**: "입력 텍스트 레이어 품질과 무관하게 전 페이지를 이미지화해 동일 경로로 처리합니다. 레이아웃·OCR·
표구조 검출 뒤 정제·요약·키워드·임베딩까지 STEP 1~8로 이어지고, 검색 시점에 RAG 생성이 붙습니다."
- **Q. CPU만으로 현실적인가?** A. 실행은 확인했습니다. 긴 논문은 동기 응답이 비현실적이라 작업 큐 배치로
  전환했고, 페이지당 지연은 실제 논문으로 재측정이 필요합니다.

### 9. 전체 시스템 아키텍처
**말할 것**: "3계층입니다 — 클라이언트, 검색·검수를 함께 처리하는 단일 API와 비동기 워커, 그리고 PostgreSQL·
Redis·로컬 LLM·임베딩으로 이뤄진 저장·모델 계층. pgvector로 정형·벡터를 한 저장소에서 다루고, 런타임에
외부 호출이 없습니다."
- **Q. 왜 전용 Vector DB가 아닌 pgvector인가?** A. 정확 매칭 SQL과 벡터 검색, 엑셀 다중 조인이 한 저장소에서
  단순합니다. 규모가 커지면 분리를 검토하되 측정 후 결정합니다.

### 10. 배포 파이프라인 (CD)
**말할 것**: "main 병합 시 클라우드 CI 테스트를 통과하면, 온프레미스 셀프호스트 러너가 deploy.sh로 이미지
빌드부터 헬스체크, 실패 시 롤백까지 자동 수행합니다. 사람 개입 없는 완전 자동 CD입니다."
- **Q. 롤백 시 DB도 되돌아가나?** A. 코드(이미지)만입니다. 이미 적용된 마이그레이션은 자동 복원하지 않으므로
  파괴적 변경은 하위 호환으로 설계합니다.

### 11. 오픈소스 · 라이선스
**말할 것**: "레이아웃·OCR은 PaddleOCR PP-StructureV3, LLM은 Qwen2.5, 임베딩은 BGE-M3, 저장소는 pgvector입니다.
상용 이용이 가능한 Apache·MIT·BSD 계열만 채택하고, CC BY-NC·GPL·AGPL 계열은 배제했습니다."
- **Q. 이 프로젝트 자체를 오픈소스로 공개하나?** A. 아니요. 여기서 '오픈소스화'는 채택한 오픈소스 기술과
  라이선스 근거를 뜻하며, 서비스 자체 공개 계획은 없습니다.

### 12. 핵심 체크포인트
**말할 것**: "세 가지입니다 — 형식 무관 안정적 추출, 검수에서 엑셀까지의 반자동 자동화, 그리고 온프레미스.
자동 검출 → 사람 검수 → 확정 적재 → 근거 기반 검색이 전부 로컬에서 돕니다."

### 13. 자동화 판단 기준과 대응
**말할 것**: "완전 자동이 아니라 반자동인데 판단 기준이 분명합니다. 블록 신뢰도로 자동/검수를 구분하고,
그림·수식·참고문헌은 규칙으로 거르며, LLM 출력은 스키마·언어 검증을 거칩니다. 기대에 못 미치면 신뢰도 낮은
영역은 사람이 보정하고, LLM 실패는 재시도·폴백·명시적 실패로 다루며 저품질은 적재하지 않습니다. 품질 바는
합격선이고, 미달한 태스크만 파인튜닝합니다."
- **Q. 신뢰도 임계값은 정해져 있나?** A. 낮은 신뢰도 영역을 검수 대상으로 표시하는 방식이며, 절대 임계는
  평가셋으로 조정할 값입니다. 지금은 사람 확인을 거치는 반자동이라는 점이 핵심입니다.

### 14. 성능 평가 기준 (합격선)
**말할 것**: "'측정 없이 튜닝 없다'가 평가 원칙입니다. 지표마다 합격선(레이아웃 mAP 0.85·CER 3%·TEDS 0.85·
F1 0.90)을 먼저 정하고 미달한 태스크만 파인튜닝하는 게이트 방식입니다. 지금은 파일럿 단계이고 정식 지표는
골든셋 완료 후 냅니다."
- → 상세 대본·Q&A는 **`EVALUATION_NOTES_AND_QA.md`** 참고.

### 15. 파일럿 평가 결과와 해석
**말할 것**: "모델이 보고하는 신뢰도는 높지만 사람 라벨과 대조하면 레이아웃은 그만큼 안 나옵니다 — 신뢰도가
정확도를 과대평가하는 거죠. OCR은 둘 다 높습니다. 레이아웃이 병목이고, 원인은 소표본·제외 경계·도메인 갭.
예비값이라는 점을 분명히 하고, 평가셋 확대 후 재측정합니다."
- → 상세 대본·Q&A(실측 여부, n=15 유의성, 스케일 등)는 **`EVALUATION_NOTES_AND_QA.md`** 참고.

### 16. 형상관리 — 버전관리와 남은 공백
**말할 것**: "git short SHA를 이미지 태그로 써서 배포 시점 코드와 이미지를 1:1로 대응시키고, 롤백은 직전 SHA로
즉시 합니다. 남은 공백은 모델·프롬프트 버전 기록 미구현 등이며, 재현성 확보의 다음 과제로 정직하게 둡니다."

### 17. 시연 시나리오
**말할 것**: "정상 흐름은 업로드 → 검수 → 적재 → 질의 → 대표·연관 + 근거 설명 → 엑셀입니다. 예외도 의도적으로
보여줍니다 — 비동기 실패 시 동기 폴백, 과부하 시 503, 언어 오염 시 재시도, 신뢰 불가 시 명시적 실패. '잘 되는
것'만이 아니라 실패를 어떻게 다루는지를 보여주는 게 목적입니다."
> 라이브 데모를 한다면 이 슬라이드에서 검수 화면 → 검색 UI → 엑셀 순으로 전환하고, 예외 중 1개는 반드시 트리거.

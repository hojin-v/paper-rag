# 2026-07-12 실제 논문 2편 OCR·RAG 실측

학습하지 않은 로컬 모델로 실제 영문 논문 2편을 전체 OCR하고, 레이아웃·단락·키워드·요약·임베딩
적재 결과를 평가한다. 실행 성공과 품질 합격을 구분하며, 사람이 교정한 값은 자동 결과로 집계하지 않는다.

```text
PDF → 200 DPI 이미지 → PP-DocLayout-M → PP-OCRv5 mobile
    → 본문 단락화 → Qwen2.5 7B 요약·키워드 → BGE-M3 임베딩
    → PostgreSQL/pgvector → 검수 viewer + 평가 Excel
```

# 1단계: 확인 자료

| 자료 | 경로 |
| --- | --- |
| 전체 평가 Excel | `outputs/evaluations/2026-07-12-two-paper-ocr-evaluation.xlsx` |
| 자동 요약 보고서 | `outputs/evaluations/2026-07-12-two-paper-ocr-evaluation.md` |
| 1번 검수 화면 | `http://localhost:8000/documents/d0c0ad3c31454faea4d4900ef97d32ea/viewer` |
| 2번 검수 화면 | `http://localhost:8000/documents/de12175eb75e409281736990c16587cd/viewer` |

Excel은 `논문요약`, `레이아웃_OCR`, `단락_요약`, `대표키워드`, `표_추출` 시트로 구성한다.
`레이아웃_OCR`은 자동 유형·좌표와 현재 유형·좌표, 원시 OCR과 교정 OCR을 나란히 저장한다. 노란색은
사람이 교정한 레이아웃/OCR이고, 붉은색은 요약 언어 또는 응답 형식 오염을 탐지한 행이다.

# 2단계: 논문별 결과

| 항목 | Bengali Document Layout Analysis | Intelligent Document Processing using AWS |
| --- | ---: | ---: |
| 페이지 | 4 | 5 |
| 자동 레이아웃 블록 | 69 | 80 |
| 평균 레이아웃 신뢰도 | 0.7873 | 0.7142 |
| 신뢰도 0.5 미만 블록 | 4 | 8 |
| 적재 본문 단락 | 35 | 26 |
| 본문 관련 판정 | 33 | 24 |
| 표 영역 | 2 | 0 |
| 대표 키워드 | 4 | 3 |
| 사람 교정 블록 | 0 | 3 |

첫 논문의 자동 레이아웃 유형은 제목 1, 초록 2, 섹션 제목 15, 본문 45, 그림 2, 표 2,
참고문헌 2개다. 제목·초록·본문·표를 실제 적재했지만 저자/소속은 본문으로 분류되어 DB 저자가
비어 있다. 그림 내부 문자가 OCR 본문에 섞인 사례와 `confi guration`, `64Opx` 같은 OCR 오자가 있다.

두 번째 논문의 자동 레이아웃 유형은 본문 52, 섹션 제목 9, 머리말/꼬리말 15, 그림 4개다. 제목과
초록을 자동으로 찾지 못했다. 자동 `text`였던 3개 영역을 검수 단계에서 제목 1개와 저자 2개로
교정했다. 제목 교정 영역의 원시 OCR은 저널 로고인 `IJFMR`이며, 실제 제목은 자동 추출 결과가 아니다.
초록은 원문에서 추출되지 않았으므로 외부 메타데이터로 채우거나 생성하지 않고 결측으로 남겼다.

# 3단계: 키워드와 요약 실측

| 논문 | 대표 키워드 |
| --- | --- |
| Bengali Document Layout Analysis | YOLOv8, Bengali Document Layout Analysis, Data Augmentation, Post-Processing |
| Intelligent Document Processing using AWS | 지능형 문서 처리, OCR, RPA |

대표 키워드는 두 논문의 중심 주제를 반영했다. 논문당 대표 키워드 3~5개만 `paper_keywords`에
연결하고, 단락별 1~3개 키워드는 `paragraphs.keywords`에 별도 저장했다. 과거처럼 모든 단락 키워드를
논문 대표 키워드로 승격하지 않는다.

단락 요약 61개 중 6개(9.8%)에서 중국어·벵골어 문자가 섞였다. 첫 논문 단락 3, 13, 16, 17, 22와
두 번째 논문 단락 8이며, 이 중 첫 논문 17과 두 번째 논문 8은 사과문·JSON 안내 같은 응답 형식 오염도
포함한다. 첫 논문 단락 22는 실제 방법 설명인데 `is_topic_relevant=false`로 잘못 판정됐다. 즉 7B
모델과 JSON 스키마만으로 출력 파싱은 가능하지만, 요약 언어 일관성과 관련성 판정을 production
품질로 보장하지 못한다.

# 4단계: 저사양 CPU 실행에서 확인한 자원 제약

| 구성 | 실측 메모리/동작 |
| --- | --- |
| Qwen2.5 7B Q4 | Ollama 컨테이너 약 4.876GiB |
| Paddle 런타임이 로드된 API | 약 1.0GiB |
| BGE-M3 서버 | 약 0.25GiB RSS |
| 현재 호스트 | RAM 7.5GiB, swap 2.0GiB |

Paddle 런타임, BGE-M3, 7B LLM을 동시에 유지하자 swap이 가득 차고 첫 7B 요청이 Ollama HTTP 500으로
실패했다. OCR 결과를 디스크에 보존한 뒤 API를 가볍게 재시작하고, `LLM 생성 → LLM unload → BGE
임베딩` 순서로 실행하면 두 번째 논문 26단락과 대표 키워드, 임베딩 저장을 완료했다. 따라서 8GB급
CPU 머신에서는 모델을 모두 상주시킨 동시 처리 구조가 아니라 작업 단계별 모델 수명 관리와 단일 작업
큐가 필요하다. 더 높은 동시성을 목표로 하면 RAM 16GB 이상 또는 별도 추론 서버가 현실적이다.

# 5단계: 숨기지 않는 production 공백

| 공백 | 이번 조치 | 남은 조치 |
| --- | --- | --- |
| 교정 후 자동 유형을 알 수 없음 | `detected_block_type`, `detected_bbox`를 별도 보존 | 변경 이력·교정자·시각 감사 로그 추가 |
| LLM 실패 재시도 시 중복 논문 행 | 불완전 paper 7 삭제 후 실패 시 논문·종속 데이터 보상 삭제 구현 | stage 7의 전역 키워드 빈도까지 원복하는 단일 트랜잭션 또는 source job 멱등키 |
| 요약의 다국어/형식 오염 | Excel 자동 경고와 원문 대조 제공 | 출력 언어 검증 후 재생성, 소형 평가셋으로 합격선 설정 |
| 제목·초록 레이아웃 누락 | 원시값 보존 후 제목/저자만 명시적 사람 교정 | 더 다양한 논문으로 클래스별 recall 측정 후 모델 변경 판단 |
| 표 구조 복원 | 첫 논문의 표 OCR 텍스트와 요약 저장 | 병합 셀·무선 표는 별도 table model 평가 필요 |
| 동기식 장시간 처리 | 단락별 LLM 캐시로 재시작 비용 감소 | 비동기 작업 큐·진행률·취소·자원 세마포어 구현 |

이번 2편만으로 파인튜닝 여부를 결정하지 않는다. 현재 모델은 학습 없이 데모 가능한 결과를 냈지만,
레이아웃 메타데이터 누락과 요약 오염이 실제 확인됐으므로 10편 이상 골든셋에서 제목/초록 recall,
본문 블록 F1, OCR CER, 요약 사실성·언어 일관성, 키워드 적합도를 측정한 뒤 프롬프트·모델 교체·학습을
순서대로 판단한다.

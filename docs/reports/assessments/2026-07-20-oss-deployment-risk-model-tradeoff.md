> **갱신(2026-07-20, 같은 날 후속 작업)**: 이 문서가 "높음" 위험·최우선 권고로 지적한 PyMuPDF
> (AGPL-3.0)는 §5 권고 ②안대로 `pypdfium2`(Apache-2.0/BSD-3-Clause)·`pypdf`(BSD-3-Clause)·

실제 배포(온프레미스 납품·폐쇄망 반입) 관점에서 오픈소스 의존성이 야기할 수 있는 문제를 점검하고,

새 실측을 수행하지 않고 `pyproject.toml`·설치된 패키지·기존 ADR/가이드/벤치마크 리포트와 라이선스
공개 정보를 교차 검증해 종합했다.

```text

├─ 1. 오픈소스 의존성 인벤토리 + 라이선스 (실측 버전 + 공식 라이선스 교차검증)

├─ 3. 모델 선택 트레이드오프 (레이아웃/OCR · LLM · 임베딩, 기존 실측 리포트 인용)
├─ 4. 종합 판단
└─ 5. 핵심 권고 (우선순위 순)
```

# 2단계: 실제 배포 시 발생 가능한 문제

## 2.1 라이선스 — PyMuPDF(AGPL-3.0)가 "폐쇄망 반입" 배포 모델과 충돌 가능 (해결됨)

- DESIGN.md §2는 배포 방식을 "폐쇄망 반입: 이미지 tar + 모델 파일 번들"로 정의한다. 이는 AGPL 관점에서
  명백한 배포(distribution)다. AGPL은 배포뿐 아니라 **네트워크로 서비스 제공(SaaS)** 시에도 완전한
  대응 소스 공개 의무가 발생하는 조항(§13)을 포함한다.
- 즉 (a) paper-rag를 고객사에 이미지로 납품하거나 (b) 사내에서 여러 팀이 네트워크로 접근하는 서비스로
  운영하는 두 시나리오 모두, PyMuPDF를 AGPL 조건 그대로 쓴다면 paper-rag 전체(또는 최소한 결합된
  실행 형태)의 완전한 소스를 그 수신자에게 공개해야 한다는 의무가 발생할 수 있다. Artifex는 이를
  피하기 위한 상용 라이선스를 별도 판매한다(연 단위 라이선스 비용 발생).
- ADR-0002에서 모델 후보에 적용한 라이선스 기준(비상업·조건부 배제)을 그대로 적용하면 PyMuPDF도
  같은 검토 대상이어야 한다. 현재 문서 어디에도 이 점이 다뤄지지 않았다.
- 대안: ① Artifex 상용 라이선스 구매, ② `pypdfium2`(Google PDFium 바인딩, Apache-2.0/BSD) 등
  permissive 렌더러로 STEP 1 교체, ③ 단독 개발·내부 실증 단계에서는 당장 문제가 아니나 **고객 납품
  시점 전에 법무 확인 필수**로 명시.
- **해결(2026-07-20)**: ②안을 채택해 PDF 렌더링은 `pypdfium2`, 페이지 복사(`collect/smoke.py`)는
  `pypdf`, 텍스트 레이어 진단 백엔드(`simple_backend.py`)는 `pdfplumber`로 교체했다(전부
  BSD-3-Clause/Apache-2.0/MIT). 테스트 픽스처용 합성 PDF 생성은 `reportlab`(BSD, dev 의존성)으로
  교체했다. 실제 22페이지 논문 PDF·200DPI 기준 실측 결과 렌더링 시간은 페이지당 약 20ms 느려졌으나
  (pypdfium2 0.069s/page vs PyMuPDF 0.050s/page) OCR 추론 시간(페이지당 분 단위)에 비해 무시할
  수준이며, 페이지 PNG 파일 크기는 약 47% 커졌다(PNG 압축 특성 차이로 추정, 추가 최적화는
  하지 않음 — 현재 디스크 용량이 제약 조건으로 명시된 바 없음). `docs/dev-log/2026-07-20.md` 참고.

## 2.2 재현성/공급망 — 버전 미고정과 락파일 부재

- `pyproject.toml` 확인 결과 `paddlepaddle==3.3.0`, `paddleocr==3.7.0`만 정확히 핀되어 있고,
  fastapi·uvicorn·sqlalchemy(`>=2.0`만)·httpx·openpyxl·numpy·sentence-transformers·docling·
  celery·streamlit은 상한 없이 오픈되어 있다.
- 락파일(`uv.lock`, `requirements*.lock`)이 저장소에 없다 — `pip install` 시점에 따라 다른 버전
  조합이 설치될 수 있다. 특히 `transformers`(4.48.3)·`sentence-transformers`(3.4.1)·`docling`은
  breaking change가 잦은 라이브러리로, 실제 docling 어댑터가 한 차례 `export_to_markdown()` 방식
  변경으로 표·참고문헌 블록 타입이 유실된 이력이 있다(2026-07-04 벤치마크 리포트).
- "폐쇄망 반입" 특성상 현장에서 인터넷 접근 없이 재설치·재현해야 하는데, 락파일이 없으면 개발
  환경과 다른 조합이 현장에 깔릴 위험이 실질적이다.
- 권고: `uv lock` 또는 `pip-compile`로 전체 의존성 트리를 고정하고, 폐쇄망 반입 시 wheel 캐시를
  이미지에 함께 번들.

## 2.3 폐쇄망 네트워크 접근 — 조용한 외부 연결 시도

- PaddleX·HuggingFace Hub 계열 라이브러리는 로컬 캐시가 없으면 기본적으로 네트워크 다운로드를
  시도한다. `scripts/download_paddle_models.py`로 Paddle 3종은 사전 다운로드하지만, `BAAI/bge-m3`
  로딩 경로(`sentence-transformers`)도 HF Hub 캐시 부재 시 같은 문제(폐쇄망에서 긴 타임아웃 대기,
  혹은 인터넷이 우연히 열린 환경에서 조용히 최신 리비전을 당겨오는 재현성 오염)를 일으킬 수 있다.
- 권고: `HF_HUB_OFFLINE=1`, `TRANSFORMERS_OFFLINE=1`을 운영 프로파일 환경변수로 명시하고
  `scripts/preflight.py`에서 오프라인 강제 여부를 검증 항목에 추가.

## 2.4 보안

- pypdfium2·PaddleOCR 모두 신뢰되지 않은 사용자 업로드 PDF를 직접 파싱한다. PDF는 대표적인 공격
  벡터(압축 폭탄, 손상된 스트림으로 인한 크래시, 임베디드 스크립트)이며, 이는 이미 `docs/guide/
  10-production-readiness.md`에 "인증·권한·TLS·악성 PDF 격리 없음"으로 기록된 리스크와 일치한다.
  `PAPERRAG_PADDLE_ISOLATE_PROCESS`로 프로세스 분리는 되어 있으나 seccomp·리소스 제한 수준의
  샌드박스는 아니다.
- `docker-compose.yml`에서 Redis(6379)·Ollama(11434) 포트가 인증 없이 호스트에 노출된다. 내부망
  전제라도 바인딩 제한·인증 계층 없이 그대로 배포되면 임의 작업 주입(Redis)이나 모델 조작(Ollama
  API는 기본적으로 인증이 없다)이 가능하다. 다중 사용자·외부 노출 단계로 갈 경우 프록시 뒤 인증
  계층이 선행 조건이다.

## 2.5 운영 안정성/리소스 — 무거운 오픈소스 모델의 동시 상주 문제

이미 실측된 잔여 위험을 종합하면, 오픈소스 라이브러리 자체 결함이라기보다 **무거운 모델 3종(Paddle,
BGE-M3, 7B LLM)을 저사양 단일 서버에 동시 상주**시키는 아키텍처 선택에서 비롯된 문제다.

| 실측 사실 | 출처 |
| --- | --- |
| 7.5GiB RAM 머신에서 Paddle+BGE-M3+7B 동시 상주 시 swap 가득 참, Ollama 첫 요청 HTTP 500 | 2026-07-12 두 논문 실측 리포트 |
| 200 DPI 2단 논문 1페이지 동기 처리 약 5~6분 | 2026-07-12 production-readiness 리포트 |
| FileReviewStore가 로컬 JSON 파일 — 다중 API replica 동시 수정 안전성 없음 | `docs/guide/10-production-readiness.md` |

스케일 아웃(동시 사용자·동시 논문 처리)을 목표로 하면 RAM 16GB 이상 또는 임베딩/LLM을 별도 추론
서버로 분리하는 재설계가 필요하다는 점이 실측으로 이미 확인되어 있다.

## 2.6 유지보수/커뮤니티 성숙도

- **Docling**: MIT, 활발하지만 API 변경이 잦다(어댑터가 breaking change로 표·참고문헌 타입을
  유실한 이력, 2026-07-04). 현재 운영 미사용(진단 전용)이라 즉각적 리스크는 낮지만, 향후 다시
  운영 경로로 편입할 계획이 있다면 버전 고정이 필수다.
- **PaddleOCR/PaddleX**: Baidu 주도 대형 프로젝트, Apache-2.0으로 활발히 유지되지만 네이티브
  런타임(MKLDNN 등) 의존성이 커서 `libgomp1` 부재로 인한 우회(`.vendor/libgomp1`, MKLDNN 비활성화)가
  이미 한 차례 필요했다 — 배포 대상 OS마다 재현되지 않을 수 있는 네이티브 의존성 리스크.
- **kiwipiepy**: 개인 유지보수(bab2min) 프로젝트, LGPL-3.0. 활발하지만 단일 메인테이너 bus-factor
  리스크가 있다 — 장기 지원 계획에서 대체 후보(예: mecab-ko)를 예비로 검토해 둘 가치가 있다.

# 3단계: 모델 선택 트레이드오프

## 3.1 레이아웃·OCR — PP-DocLayout-M + PP-OCRv5 vs Docling vs 배제 후보

| 후보 | 라이선스 | 강점 | 실측된 약점 | 채택 여부 |
| --- | --- | --- | --- | --- |
| PP-DocLayout-M + PP-OCRv5(현 채택) | Apache-2.0 | 디지털·스캔 단일 경로, PaddleX 파인튜닝 공식 지원, CPU 구동, 좌표·신뢰도 보존 | 평균 레이아웃 신뢰도 0.71~0.79(실 논문 2편), 두 논문 모두 제목/초록 완전 자동 추출 실패, 저자·소속 오분류로 DB 저자 필드 결측, 표는 병합 셀·복잡 무선 표 구조 미보장 | 채택(MVP) — 파인튜닝 게이트 미도달 |
| Docling | MIT | TableFormer 표 인식 우수, 섹션 보존 단락 분리 우수(합성 PDF 8 단락 vs simple 3) | 스캔 PDF 미대응, 초기 어댑터 버그로 표/참고문헌 타입 유실 이력, 합성 PDF에서 제목 라벨 미분류 관찰 | 진단/비교 전용으로 격하(2026-07-12 사용자 결정: 전체 OCR 단일 경로 확정) |
| MinerU | AGPL-3.0 | 벤치마크 참고 성능 우수 | 라이선스 배제 | 미채택(baseline 전용) |
| DocLayout-YOLO | AGPL-3.0 | — | 라이선스 배제 | 백업 보류 |
| LayoutLMv3/DiT | 가중치 CC BY-NC | — | 비상업 배제 | 미채택 |
| Surya/Marker | GPL + 가중치 조건부 | — | 배제 | 미채택 |

**판단**: 라이선스 기준 선별(ADR-0002)은 근거가 탄탄하고 적절하다. 다만 실측 품질(레이아웃 신뢰도
0.71~0.79, 제목/초록 recall 저조)은 DESIGN §6 게이트(클래스별 mAP@0.5 ≥0.85 등) 대비 아직
측정되지 않은 상태다. 즉 "모델 선택 자체"는 라이선스·구조 면에서 적절하지만, "production 품질
합격"은 아직 주장할 수 없는 조건부 적절 단계다.

## 3.2 LLM — Qwen2.5-7B-Instruct Q4(Ollama) vs 1.5B vs 상용 API

| 후보 | 라이선스/데이터 위치 | 실측 근거 | 트레이드오프 |
| --- | --- | --- | --- |
| Qwen2.5-7B Q4(현 채택) | Apache-2.0, 완전 온프레미스 | CPU 경합 상태에서 900초 타임아웃 2회 → 패스스루 폴백(2026-07-04); 7.5GB RAM 환경 swap thrashing으로 첫 요청 HTTP 500(2026-07-12); 실 논문 61단락 중 9.8%에서 다국어(중국어·벵골어) 오염, 방법론 단락 1건 관련성 오탐 | 온프레미스·라이선스 요건은 충족하나 저사양 CPU에서 지연·품질 모두 production 미달로 실측 확인됨 |
| Qwen2.5-1.5B | Apache-2.0 | 219.6초 응답, 영한 혼용·의역 오류("스mart factory", "디지털 시대의 공장") 관찰 | 개발 머신 기능 검증 전용으로 이미 판단됨, 운영 투입 불가 |
| 상용 API(GPT/Claude 등) | 폐쇄형, 데이터 외부 전송 | 미검증(설계상 배제) | "온프레미스 논문 분석 RAG"라는 프로젝트 전제(논문 원문 외부 전송 금지) 자체와 정면 충돌 — 후보에서 제외하는 것이 구조적으로 타당 |

**판단**: 온프레미스 제약에 부합하는 선택으로 구조적 적절성은 있다. 그러나 CPU 추론의 속도-크기
트레이드오프가 아직 해소되지 않았다 — 7.5GB RAM 개발 머신 실측은 스스로도 "품질 판단에 사용하지
않는다"고 명시한 조건이며, DESIGN §6가 요구하는 "16코어급 서버 또는 GPU 재측정" 게이트가 열려
있다. 현재는 모델 선택 자체보다 **배포 목표 하드웨어 사양 미확정**이 더 실질적인 리스크다.

## 3.3 임베딩 — BGE-M3

| 후보 | 라이선스 | 트레이드오프 |
| --- | --- | --- |
| BGE-M3(현 채택) | MIT | 1024차원, 100개 이상 언어의 한/영 혼용 성능 우수, CPU 추론(로컬 캐시 4.3GB) 실측 성공. 다만 유사어 쌍 판정 임계값을 0.6→하향 조정한 이력이 있음(실측에서 "예지보전 ↔ 예측 유지보수" 유사도 0.59 관찰) — 파라미터가 아직 소규모 데이터로만 보정된 상태 |
| OpenAI/Cohere 임베딩 API | 폐쇄형 | 미검증, 온프레미스 제약과 구조적으로 충돌해 배제 타당 |

**판단**: MIT 라이선스로 법적 리스크가 없고 온프레미스 제약에 부합해 적절하다. 다만 임계값 등
세부 파라미터가 소규모 실측(유사어 쌍 극소수)으로만 보정되어 있어, 결정 자체보다 "보정 데이터
양 부족"이 남은 리스크다.

# 5단계: 핵심 권고 (우선순위 순)

1. ~~**(배포 전 필수) PyMuPDF AGPL-3.0 노출 검토**~~ — **해결됨(2026-07-20)**: `pypdfium2` 등
   permissive 렌더러로 교체 완료(§2.1 참고). Artifex 상용 라이선스 구매나 AGPL 공개 검토는
   더 이상 필요하지 않다.
2. **의존성 락파일 도입**(`uv lock` 또는 `pip-compile`) — 폐쇄망 반입 시 재현 가능한 빌드를 보장하고,
   `transformers`/`sentence-transformers`/`docling`처럼 API 변경이 잦은 라이브러리의 breaking
   change를 배포 시점에 차단한다.
3. **Redis 이미지 태그 고정 또는 전환 결정** — `redis:7-alpine`이 실제로 어떤 라이선스(RSALv2/SSPLv1
   vs 이전 BSD 계열)의 리비전을 받는지 명시적으로 확인하고, 필요 시 BSD 계열 태그 고정 또는
   Valkey·Redis 8.x(AGPLv3 옵션)로 전환한다.
4. **LLM을 운영급 서버(16코어/64GB 또는 GPU)에서 재실측** — DESIGN §6 품질 게이트 판정에 사용한다.
   9.8% 다국어 오염은 이미 production 임계를 넘는 수치이므로 프롬프트 보강 또는 모델 교체 결정이
   필요하다.
5. **오프라인 강제 환경변수**(`HF_HUB_OFFLINE=1`, `TRANSFORMERS_OFFLINE=1`)를 운영 프로파일과
   `scripts/preflight.py` 검증 항목에 추가해 폐쇄망에서의 조용한 네트워크 접근 시도를 차단한다.
6. **kiwipiepy 대체 후보 예비 조사**(예: mecab-ko) — 당장 조치는 아니나 단일 메인테이너 bus-factor
   리스크를 장기 지원 계획에 기록해 둔다.

# 2026-07-22 검색 LLM 전환에 따른 서버 사양·비용 분석

2026-07-22 결정으로 검색 키워드 추출을 형태소 분석(Kiwi) 기본 경로에서 LLM(Ollama) 상시 경로로
전환하고, 대표/연관 논문마다 관련도 설명(RAG 생성 단계)을 추가했다. 이 문서는 그 결정이 실제로
얼마나 지연·자원 부담을 늘리는지 실측치로 계산하고, "실사용 가능한" 수준별로 필요한 서버 사양을
정리한다. 추측이 아니라 이 서버(운영 환경 그 자체)에서 확인한 수치만 근거로 쓴다.

## 1단계: 현재 실측 기준선

### 하드웨어

이 저장소가 곧 운영 서버다(Tailscale IP `100.80.39.49`가 이 맥북의 `utun4` 인터페이스에 그대로
잡혀 있음, `docker-compose.yml`의 `env_file: .env`가 이 리포지토리의 `.env`를 그대로 읽음).

| 항목 | 값 |
| --- | --- |
| CPU | Intel Core i7-9750H (6코어/12스레드, 2.60GHz) |
| 물리 RAM | 16GB |
| **Docker Desktop VM 가용 RAM** | **7.75GiB** (`docker stats`로 전 컨테이너 공통 확인) |
| GPU | Intel UHD Graphics 630(내장, VRAM 1.5GB) — CUDA/Metal 가속 추론 불가 |
| OS | macOS 26.5 |

**중요한 함정**: 물리 RAM은 16GB지만 Postgres·Redis·Ollama·API·worker가 모두 Docker Desktop의
Linux VM 안에서 도는데, 이 VM에 할당된 메모리가 **7.75GiB뿐**이다(`docker stats` 전 컨테이너의
`MEM USAGE / LIMIT`이 공통으로 7.75GiB). 2026-07-12 작업 일지가 기록한 "swap이 가득 차 요청이
실패" 사고는 이 7.75GiB 안에서 Paddle OCR + BGE-M3 + Qwen2.5 7B가 동시에 상주하려다 벌어진
일이지, 16GB 전체가 부족했던 게 아니다.

### LLM 실측치

| 항목 | 값 | 근거 |
| --- | --- | --- |
| 모델 | `qwen2.5:7b-instruct-q4_K_M` | `docker exec ... ollama list` (이 서버에서 직접 확인, 4.7GB) |
| 호출 1건 지연 | 18~20초 | `search/service.py`/`ui/app.py` 주석의 실측치(짧은 프롬프트·최대 192 토큰 출력 기준) |
| 동시 요청 처리 | 완전 직렬 | `heavy_task_max_concurrency=1`(Redis 세마포어, `concurrency.py`) |
| 세마포어 대기 한도 | 60초 | `heavy_task_semaphore_wait_seconds` — 초과 시 `HeavyTaskBusyError`(503) |

`heavy_task_max_concurrency=1`은 우연이 아니라 위 스와핑 사고 이후 **의도적으로** 못박은 값이다
(`concurrency.py` 모듈 docstring). 검색 시점 LLM 호출도 동일한 `heavy_task_slot()`을 거치므로
(`llm_enrich.py:173`), 지금은 **검색 요청과 논문 적재(OCR) 요청이 같은 슬롯 1개를 두고 경쟁**한다.

## 2단계: 이번 변경으로 늘어난 부하량

검색 1건이 유발할 수 있는 LLM 호출 수:

| 상황 | LLM 호출 수 | 최악 지연(18~20초 기준) |
| --- | --- | --- |
| 이미 검색된 키워드(캐시 히트) | 0회 | 즉시 (DB 조회만) |
| 처음 검색하는 키워드, 연관 논문 없음 | 2회(키워드 추출 + 대표 논문 관련도 설명) | 36~40초 |
| 처음 검색하는 키워드, 연관 논문 있음 | 3회(키워드 추출 + 대표·연관 관련도 설명 각 1회) | 54~60초 |

`keyword_result_cache`(기존)와 이번에 추가한 관련도 설명 재사용 로직(`_relevance_summary`가
캐시에 이미 있으면 LLM을 다시 부르지 않음) 덕분에 **같은 키워드의 두 번째 검색부터는 LLM 호출이
0회**다. 즉 비용은 "키워드당 최초 1회"로 상한선이 있고, 무한정 누적되지 않는다. 다만 논문
데이터베이스가 커질수록 "처음 보는 키워드"의 절대 수는 늘어난다.

동시 사용자 영향: `heavy_task_max_concurrency=1`이므로 사용자 A가 새 키워드를 검색하는 동안
(최악 60초) 사용자 B가 역시 새 키워드를 검색하면, B는 세마포어 대기 한도(60초)에 걸려 **`503
HeavyTaskBusyError`를 받을 수 있다** — 지금 이 사양으로는 "동시에 여러 명이 새로운 검색을
한다"는 시나리오 자체가 성립하기 어렵다.

## 3단계: 시나리오별 필요 사양

### 시나리오 A — 데모/1인 사용 (지금 그대로 실사용 가능)

발표·시연처럼 한 번에 한 사람만 쓰고, 같은 키워드를 반복 조회하는 경우라면 지금 사양으로 충분하다.
첫 검색만 수십 초 걸리고 이후는 캐시로 즉시 응답한다.

- **권장 조치(비용 0원)**: Docker Desktop의 VM 메모리 할당을 7.75GiB → 12GB 이상으로 늘린다.
  물리 RAM 16GB 중 macOS 자체 사용분을 남기고 대부분을 VM에 줄 수 있다. 이것만으로 스와핑 위험이
  크게 줄고, `heavy_task_max_concurrency`를 1→2로 조심스럽게 올려볼 여지가 생긴다(다만 2로 올린
  뒤에는 반드시 2026-07-12와 같은 방식으로 재측정하고 나서 반영해야 한다 — 지금 그 재측정은
  안 되어 있다).

### 시나리오 B — 팀 내 동시 사용자 5~10명

이 하드웨어(CPU 전용, 6코어)로는 어렵다. `heavy_task_max_concurrency`를 억지로 늘려도 CPU
추론은 스레드를 늘린다고 선형으로 빨라지지 않고(코어를 나눠 쓰므로 개별 요청이 더 느려짐), 이미
7.5GB급에서 스와핑을 겪은 이력이 있어 동시 다중 추론은 근본적으로 GPU 없이는 위험하다.

- **필요 조치**: GPU 1장 추가(아래 시나리오 C 참고). 굳이 CPU만 고집한다면 경량 모델(1.5B~3B)로
  교체해 호출당 지연을 낮추는 것이 차선책이지만, 이 저장소의 자체 벤치마크
  (`docs/reports/benchmarks/2026-07-04-llm-cpu.md`)에서 1.5B는 한중 혼용·의역 오류가 확인된
  바 있어 품질 리스크가 있다(그 벤치마크는 이 서버가 아니라 별도 WSL2 개발 머신 실측이라
  이 서버에서 재측정 없이 그대로 신뢰하면 안 된다).

### 시나리오 C — 실서비스 수준(다수 동시 사용자)

GPU 서버로 Ollama(또는 vLLM 등 배치 추론 서버)를 분리하는 것이 현실적인 유일한 해법이다.

- **VRAM 요구량**: `qwen2.5:7b-instruct-q4_K_M` 파일 자체가 4.7GB. 컨텍스트·배치 오버헤드를
  더하면 실사용에는 8GB 이상 VRAM을 권장한다 — RTX 4060 Ti 16GB, RTX 3060 12GB급이면 여유 있게
  돌아간다.
- **기대 효과**: 짧은 프롬프트(질의 키워드 추출·관련도 설명 모두 출력 192 토큰 이하) 기준으로
  GPU 추론은 통상 1~3초대 응답이 기대된다(18~20초 대비 10배 이상 개선) — 다만 이 수치는 이
  서버에서 GPU로 실측한 값이 아니라 일반적인 7B Q4 GPU 추론 특성에 근거한 추정이므로, 실제
  도입 전 반드시 후보 GPU에서 재측정해야 한다.
- **동시성**: Ollama 자체는 `OLLAMA_NUM_PARALLEL` 설정으로 제한적 동시 처리를 지원하지만, 진짜
  다중 동시 사용자를 노린다면 연속 배치(continuous batching)를 지원하는 vLLM/TGI 같은 서빙
  스택으로 교체하는 것이 더 확실하다 — 이는 이번 분석 범위를 넘는 별도 검토가 필요하다.
- **온프레미스 원칙과의 관계**: 이 프로젝트의 설계 전제(`DESIGN.md`)는 폐쇄망 온프레미스 운영이다.
  GPU 서버를 추가해도 "사내에 두는 별도 장비"이므로 원칙과 충돌하지 않는다.

## 4단계: 대안 비교

| 대안 | 비용 | 효과 | 온프레미스 원칙 |
| --- | --- | --- | --- |
| Docker VM 메모리 증설(7.75→12GB+) | 0원 | 스와핑 위험 감소, 동시성 1→2 여지 | 문제없음 |
| 경량 모델(1.5B~3B)로 교체 | 0원 | 지연 감소 기대(이 서버 미실측), 품질 저하 위험 확인됨 | 문제없음 |
| GPU 서버 추가(Ollama 분리) | 하드웨어 구매(GPU 1장) | 지연 10배+ 개선, 진짜 동시성 확보 | 문제없음(사내 장비) |
| 클라우드 LLM API로 대체 | 종량 과금 | 가장 쉽고 빠름 | **원칙 위반** — 이 프로젝트 범위에서 배제 |

## 5단계: 권장 조치

1. **즉시(비용 없음)**: Docker Desktop VM 메모리를 12GB 이상으로 늘리고, 2026-07-12와 동일한
   방식으로 Paddle+BGE-M3+Ollama 동시 상주 재측정 → 안전하면 `heavy_task_max_concurrency=2` 검토.
2. **결정 필요**: "실사용"의 기준이 몇 명 동시 사용자·몇 초 응답인지 확정해야 시나리오 B/C 중
   어느 쪽을 택할지, GPU 사양을 얼마로 잡을지 정할 수 있다. 이 문서는 시나리오별 요구 사양의
   틀만 제공하며, 목표 동시 사용자 수는 프로젝트 쪽에서 정해야 하는 값이다.
3. GPU 도입을 검토하게 되면, 후보 GPU에서 이 서버와 동일한 프롬프트로 재실측해 이 문서의
   "기대 효과" 수치를 실측치로 갱신한다.
